    Per D-01, do NOT add a dedicated new “backup status” API endpoint. Instead, extend an already-used admin fetch. The Admin UI already calls `GET /api/admin/settings` during `loadAdminData()`; extend that response to include a new field like:
    - `storage: { backend: "s3" | "disabled", location?: { bucket, key, region, endpoint }, backupStatus?: { lastSuccessAt, lastFailureAt, failureCode, failureSummary } }`
    Keep existing `settings` response shape intact.

    In `src/server/storage/s3-storage.ts`, add helpers that (a) return non-secret config details and (b) update persisted status:
    - `getS3LocationDetails()` returning `{ bucket, key, region, endpoint }` without credentials.
    - `writeBackupStatus(sqlite, patch)` that upserts into `storage_backup_status` for backend `s3`. This must be best-effort (do not throw from backup/restore handlers if status write fails).
    - `readBackupStatus(sqlite)` returning the row or `null`.

    Update the admin backup route (`POST /api/admin/backup`) to:
    - On entry: compute location details once (safe).
    - On success: write `last_success_at` and clear failure fields; include location details; update `updated_at`.
    - On failure: keep `last_success_at` unchanged; write `last_failure_at`, `failure_code="backup_fehlgeschlagen"`, and a safe `failure_summary` derived from the caught error:
      - MUST NOT include secrets, raw headers, or stack traces (D-14).
      - SHOULD include useful operator hints like “S3 credentials missing”, “snapshot empty”, “NoSuchKey”, etc if available via `error.name` / `error.message` but sanitized to a short summary.

    Enforce D-14 by construction:
    - Implement `toSafeBackupFailureSummary(err): string` (or equivalent) that builds a summary from **allowlisted fields** only (e.g. `name`, `code`, short `message`), applies **max length** (e.g. 240 chars), and explicitly forbids including `err.stack`, request/response headers, or any credential-like substrings.
    - Use this helper for BOTH (a) the persisted `failure_summary` and (b) any immediate UI error text derived from server responses.
    - Ensure the API still returns stable German-style error codes (constraint): `500 { error: "backup_fehlgeschlagen" }`.

    Update `src/main.tsx` Admin UI storage section to show a status panel (per D-02):
    - last success time (or “—”)
    - last failure time, failure code/category, and failure summary
    - non-secret location details
    This is display-only wiring; do not add a new fetch call. Use the existing admin settings load.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → admin API | Untrusted network input; admin session is still a web client. |
| server → S3 | External service interaction; errors can include sensitive request context. |
| server → SQLite | Persists operator-visible status; must not store secrets. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|----------|----------|-----------|-------------|-----------------|
| T-03-01 | I | Error/status responses | mitigate | Enforce safe summaries (no stack traces/secrets/SDK headers) and only return non-secret location details (D-02, D-14). |
| T-03-02 | R | Audit/status reliability | accept | Status writes are best-effort; primary backup action must not fail due to status persistence issues (D-03). |
| T-03-03 | T | Admin UI integrity | mitigate | Keep status sourced from server-side persisted row; do not allow client-provided overrides. |
</threat_model>
