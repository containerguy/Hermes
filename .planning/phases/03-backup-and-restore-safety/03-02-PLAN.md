    In `src/server/storage/s3-storage.ts`, introduce a structured error type for restore failures that can carry safe diagnostics without leaking internals. Example:
    - `class RestoreValidationError extends Error { diagnostics: RestoreDiagnostics }`
    - `type RestoreDiagnostics = { kind: "validation_failed" | "copy_failed" | "recovery_failed"; snapshot?: {...}; live?: {...}; missingTables?: string[]; columnMismatches?: Array<{ table: string; missingInSnapshot: string[]; extraInSnapshot: string[] }>; foreignKeyFailures?: Array<{ table: string; rowid: number; parent: string; fkid: number }>; migrations?: { liveLatest?: string; snapshotLatest?: string; liveCount?: number; snapshotCount?: number }; summary: string }`

    - snapshot location details (bucket/key/region/endpoint) but never credentials
    - migrations/version signal (latest `schema_migrations.name` in snapshot vs live) per D-11
    - table/column details for incompatibilities

    Enforce D-11/D-14 with a concrete boundary:
    - Implement `toSafeRestoreDiagnostics(error): RestoreDiagnostics` (or equivalent) that builds diagnostics via **allowlisted fields only** (no raw `error` serialization), **caps list sizes** (e.g. first 20 FK failures / first 20 mismatches), and **truncates strings** (e.g. `summary` max 240 chars). It must never include `stack`, request/response headers, or any credential-like values.
    - Ensure both the 400 validation path AND the 500 unexpected-error path go through this sanitizer before returning diagnostics to the UI.

    Implement concrete validation approach (required by prompt; must be deterministic):
    - Download snapshot to a temp file (already done).
    - Open snapshot using a temporary `better-sqlite3` connection (separate from live) OR use `ATTACH` to a validation-only connection, but do NOT mutate live tables.
    - Snapshot checks:
      1) `sqlite_master` contains expected tables:
         - Required set: the existing `restorableTables` list in `s3-storage.ts` (users, sessions, audit_logs, invite_codes, ... including `schema_migrations`). Validation fails if any required table is missing. (This is stricter than current “skip missing table” behavior, per D-04.)
      2) Schema migrations/version signals:
         - Ensure `schema_migrations` exists and is readable.
         - Read latest migration name in both live and snapshot (`SELECT name FROM schema_migrations ORDER BY name DESC LIMIT 1`).
         - Validation fails if snapshotLatest !== liveLatest (fail-safe; provides strong operator signal).
      3) Column compatibility per table (D-06):
         - For each required table, compute `liveColumns` via `PRAGMA table_info(table)` on live and `snapshotColumns` on snapshot.
         - Define `missingInSnapshot = liveColumns - snapshotColumns`. If non-empty for any table, validation fails with diagnostics listing missing columns.
         - Extra columns in snapshot are allowed and ignored during copy.
      4) Foreign key integrity of snapshot (D-05):
         - Enable FK checks in snapshot connection (`PRAGMA foreign_keys = ON`) and run `PRAGMA foreign_key_check;`.
         - Validation fails if any rows are returned; include a capped list (e.g. first 20) in diagnostics.
      5) Optional but recommended: `PRAGMA integrity_check;` in snapshot connection. If not `ok`, fail validation with a summary.

    - Catch validation errors and return `400` with `error: "restore_fehlgeschlagen"` and `diagnostics`.
    - Catch unexpected errors and return `500 { error: "restore_fehlgeschlagen" }` with sanitized `diagnostics` produced by `toSafeRestoreDiagnostics(...)` (no internals).

    Add/extend a focused test in `src/server/http/app-flow.test.ts` that asserts:

  <action>
    Choose retention count N per D-09. Use **N = 10** recoveries:
    - Justification: supports multiple operator attempts/rollbacks during an incident while keeping S3 cost bounded and cleanup simple.

    Implement pre-restore recovery snapshot creation (D-07, D-08):
    - Add `createRecoverySnapshot(sqlite): Promise<{ recoveryId: string; key: string; location: { bucket, region, endpoint } }>` in `s3-storage.ts`.
    - Key format MUST be: `recoveries/<timestamp>-<id>.sqlite` where:
      - `<timestamp>` is ISO-like but filesystem-safe (e.g. `2026-04-16T12-34-56Z`) OR epoch millis; ensure no `:` to keep S3 key readable in logs.
      - `<id>` is a short ID (8-12 chars) derived from UUID (D-08 “short recovery ID”).
    - Upload the live DB file (after `wal_checkpoint(TRUNCATE)`) to this recovery key using `PutObjectCommand`.
    - This must happen AFTER validation passes and BEFORE any destructive mutation of live tables.

    Implement recovery retention cleanup (best-effort; must not block restore):
    - Add S3 listing + deletion support using AWS SDK v3 commands:
      - `ListObjectsV2Command` with prefix `recoveries/`
      - `DeleteObjectCommand` for older keys beyond the newest N (by `LastModified`)
    - Cleanup runs after recovery upload succeeds (or at the end of restore), and failures are logged but do not fail restore.

    Implement all-or-nothing restore with explicit compatible columns only (D-06, D-10):
    - After validation succeeds and recovery snapshot is uploaded, perform the live restore in a single `better-sqlite3` transaction.
    - Implementation outline using existing `ATTACH DATABASE` approach:
      1) `ATTACH DATABASE ? AS restore_snapshot` pointing at downloaded snapshot file.
      2) Inside a transaction:
         - `PRAGMA foreign_keys = OFF` at start (required for bulk replace).
         - For each required table in restore order (use existing `restorableTables` list):
           - Compute `columnsToCopy` = `liveColumns` (from PRAGMA) (since validation ensured snapshot has all live columns).
           - Execute `DELETE FROM "table";`
           - Execute `INSERT INTO "table" ("c1","c2",...) SELECT "c1","c2",... FROM restore_snapshot."table";`
             - This MUST be explicit columns; no `SELECT *` anywhere (D-06).
         - Re-enable FK checks: `PRAGMA foreign_keys = ON`.
         - Run `PRAGMA foreign_key_check;` on the live DB and fail (throw) if any rows are returned (include capped FK failures in diagnostics).
      3) `DETACH DATABASE restore_snapshot` in a `finally` block and ensure `foreign_keys` is ON at the end.
    - Any thrown error inside the transaction must roll back and leave live DB unchanged (D-10).

    Update `POST /api/admin/restore` response on success to include recovery info (D-08):
    - `200 { ok: true, message: "restore_abgeschlossen", recovery: { id: "<short>", key: "recoveries/..." }, restoredFrom: { bucket, key } }`
    On failure, include `diagnostics` as implemented in task 03-02-01.
  </action>
