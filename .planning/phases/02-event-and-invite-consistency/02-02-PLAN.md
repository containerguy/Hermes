    Per decisions D-05..D-09 and D-08 retry behavior:

    - Pattern: follow `.planning/phases/02-event-and-invite-consistency/02-PATTERNS.md` (“event-routes.ts capacity race” + “idempotent upsert pattern”) and move only the joined-capacity read into the same txn as the upsert.
    - In the participation mutation route (the handler that currently does `countJoined(context, event.id)` before the upsert), move the capacity check into the same SQLite transaction as the participation write.
    - Use a `better-sqlite3` immediate transaction via `context.sqlite.transaction(() =&gt; { ... }).immediate()` so the write lock is acquired up front.
    - Implement exactly one bounded retry for transient SQLite contention:
      - If the immediate transaction throws `SqliteError` with `.code` matching `SQLITE_BUSY`, `SQLITE_BUSY_*`, `SQLITE_LOCKED`, or `SQLITE_LOCKED_*`, retry the whole operation once.
      - If the second attempt fails with the same busy/locked condition, return the existing error behavior used elsewhere for DB failures (do not invent a new error code).
    - Inside the transaction:
      - Load the current participation row for `(eventId, userId)` (as the handler does today) to know whether the user is already joined (idempotent join).
      - If the request is `status="joined"` and the user is NOT already joined:
        - Count joined participations for the event (`status='joined'`) inside the transaction.
        - If `joinedCount &gt;= event.maxPlayers`, throw a sentinel error that maps to `409 { error: "event_voll" }` (per D-05).
      - Perform the existing `insert(...).onConflictDoUpdate(...)` upsert inside the same transaction.
      - If the route recalculates event status based on joinedCount, keep using the existing domain logic (avoid inventing new status rules).
    - Response behavior:
      - Keep `409 { error: "event_voll" }` as the stable error code (per D-05).
      - D-09 is discretion: you MAY return `{ error: "event_voll", event: ... }` on 409, but if you do, ensure the client can still work when only `{ error }` is returned.
  </action>
