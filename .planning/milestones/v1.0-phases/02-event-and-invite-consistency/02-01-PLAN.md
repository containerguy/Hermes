  <files>src/server/http/auth-routes.ts</files>
  <action>
    Per decisions D-01..D-04 and D-03 retry behavior:

    - Pattern: follow `.planning/phases/02-event-and-invite-consistency/02-PATTERNS.md` (“auth-routes.ts transaction wrapper” + “Best-effort audit pattern”), replacing only the capacity check placement.
    - In the `POST /api/auth/register` invite-registration path, move the invite capacity check (`uses.length >= invite.maxUses`) into the same SQLite transaction that writes `users` + `invite_code_uses`.
    - Use a `better-sqlite3` immediate transaction via `context.sqlite.transaction(() =&gt; { ... }).immediate()` so the write lock is acquired up front.
    - Implement exactly one bounded retry for transient SQLite contention:
      - If the immediate transaction throws `SqliteError` with `.code` matching `SQLITE_BUSY`, `SQLITE_BUSY_*`, `SQLITE_LOCKED`, or `SQLITE_LOCKED_*`, retry the whole operation once.
      - If the second attempt fails with the same busy/locked condition, return the existing error behavior used elsewhere for DB failures (do not invent a new error code).
    - Inside the transaction:
      - Re-load the invite row by code (or by `invite.id` if already loaded) to ensure you’re checking current state.
      - Count `invite_code_uses` for that invite (do not count outside the txn).
      - If exhausted: throw a sentinel error (local to the handler) that maps to `403 { error: "invite_ausgeschoepft" }` (per D-01).
      - If available: perform user insert + invite_code_uses insert as today, and compute `remainingUses` as `maxUses - (usesAfterInsert)` when `maxUses` is non-null.
    - Response behavior:
      - Keep existing success response shape, but you MAY add `remainingUses` on success (per D-04). Do not add it on failure.
      - Keep failure response exactly `403 invite_ausgeschoepft` when exhausted (per D-01).
    - Rejection auditing:
      - When the exhausted-invite sentinel is hit, write a best-effort audit entry (per D-02) that includes IP/username in metadata if available in the request/context.
      - Use `tryWriteAuditLog` (not `writeAuditLog`) so audit failures cannot break the response.
      - Do not leak the raw invite code; use existing masking utilities if any audit metadata includes code-like strings.
  </action>

    - Rejection is audit-logged best-effort with IP/username metadata when available (per D-02).
